Introduction

When developing your professional products and projects, it is almost certain that, as a project manager, you have been responsible for controlling the quality of your products. Many might unknowingly conflate quality control with quality management, but the two are separate beasts that you must conquer throughout development. You might be asking yourself, “what is the purpose of quality control?” Well, there are several primary goals exclusive to it, including acceptance decisions, rework, and process adjustments.

Acceptance Decisions

As you develop your products and steer the wheel of your team’s ship, there are numerous decisions you’ll be making; in the context of quality control, acceptance decisions pertain to the decisions of stakeholders to either accept or reject certain project deliverables, such as features or components. This decision should be leveraged as a form of constructive criticism that can guide the team in a certain direction. If a set of related features are accepted, it is probable that those features are not lacking in the expected degree of quality, while the contrary stands true for rejected deliverables. If a deliverable or set of features end up being rejected by stakeholders, this allows the team to shift momentum and rework those features to uphold the desired level of excellence.

Rework

As indicated in the previous section, rework pertains to the actions that a team must take to refactor and possibly redesign the rejected component based on stakeholders’ feedback that will ensure a team can modify it in such a way that it complies with feature specifications, project specifications, and user/stakeholder expectations. Unfortunately, having to waste an inordinate amount of time reworking deliverables from a previous sprint allocate time away that would have been spent on prospective features, meaning that it’s essential that quality is upheld in a manner that minimizes the amount of rework to which a team must commit.

Process Adjustments

Process adjustments refer to immediate adjustments that are made to a project/product’s features based on established quality control measurements that can be obtained from a variety of techniques and tools mentioned in the following sections. Process adjustments generally entail modifying process assets and even the project management plan itself, by needing to annex external equipment/tools into the development environment in order to effectively resolve certain issues. One positive outcome of process adjustments, however, is the fact that it can not only mitigate present issues but potentially future issues that could arise on the current development path.

Tools and Techniques for Quality Control

Given the numerous features that your team may be concurrently developing for a project, it is certainly possible that improper quality control may obstruct the path to creating a high-quality product that your stakeholders want and rightfully deserve. If you are not the most experienced project manager, you are likely oblivious to many of the tools and techniques that can facilitate quality control. While countless tools exist, some helpful ones include cause-and-effect diagrams, control charts and Pareto charts.

Cause-and-Effect Diagrams

Cause-and-effect diagrams, also called fishbone diagrams, follow a simple ideology: for every problem that arises, there exists a multitude of potential root causes; by assessing the problem and the context in which it exists, you are then able to pinpoint the primary culprit and then make amends accordingly. For example, a potential problem that a stakeholder would reveal might be “unsatisfactory resource loading times” for a particular page of a new web application your team is developing. Once this problem has been identified, you can draw the problem in a box with an arrow pointing toward it; along the arrow, you will list various attributes into categories that represent potential root causes for the problem, and each cause would point to the main arrow; in this case, “poor loading times on page X” could go inside the problem box. Some problem categories might be software and hardware. Software related issues might be slow internet speed or inefficient system algorithms, and hardware related issues could include the user’s processor being too slow to quickly load the web application page. Once your diagram is set up, you can then meticulously analyze each problem set in order to ultimately determine the primary root cause so it can be targeted and resolved. Control Charts

Control charts display various statistics pertaining to a certain process and analyzing the variation in these statistics can indicate whether or not a project/product development process is under control or not. If variations are determined to be the result of minor, random causes, then a process is considered to be under control and no action needs to be taken; on the other hand, if these statistical variations are the result of possibly drastic, unexpected events, measures must be taken to modify the process in a way that allows it to regain control. Control charts can be utilized to determine potential variations in cost allocations, projected schedules, and other management-related issues arise in a process. A rule to abide by would be the “seven run rule,” which states that seven consecutive data points being all above or all below the mean may indicate that the variation and its causes should be investigated.

Pareto Charts

A Pareto chart is a type of histogram that orders various problem points by the frequency of how many complaints are received pertaining to each problem; this allows a project team to identify the most problematic issues present in a project feature or deliverable so that corrective action can be taken to alleviate said problems and potentially prevent similar issues in the future by targeting a development paradigm that conceived the most notable defects. Wrap-Up

In conclusion, it’s evident that properly controlling the quality of your project features and processes can appear to be an insurmountable precipice; however, countless tools can be leveraged in order to target problems as they arise, and also fortify your project against future problems as you modify your project’s infrastructure and development approaches. Cause-and-effect diagrams, control charts and Pareto charts are all indispensable tools, but there are many other techniques available so that you can ultimately succeed in providing a top-quality product for your consumers.
